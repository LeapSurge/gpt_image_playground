# 购买 / 兑换流程产品规格

状态：Draft v0.1  
关联文档：

## 1. 背景

当前产品的主要商业目标不是站内支付，而是让来自抖音等渠道的中国大陆用户：

- 先低门槛试用
- 试用后知道如何继续使用
- 在站外完成付款
- 通过兑换码快速恢复生成能力

当前的“邮箱 + 访问码 + 客户登录”模型更接近后台运维逻辑，不符合真实用户的购买心智。

## 2. 目标

本次改造要完成以下目标：

1. 将终端用户的心智从“登录客户账号”改为“购买 / 兑换继续使用”。
2. 保留匿名试用能力，并清楚告知试用刷新规则。
3. 在不引入站内支付和正式注册的前提下，打通最短商业闭环。
4. 尽量复用现有 customer / session / credits 结构，降低实现风险。

## 3. 非目标

本次不做：

- 站内支付
- 正式注册
- 邮箱找回
- 多设备同步
- 复杂会员中心
- 完整营销页改造

## 4. 核心假设

1. 当前阶段站外支付是合理方案，主要购买方式为：
   - 发卡网站自助购买
   - 微信沟通后转账购买
   - 抖音引流后转到购买渠道
2. 用户最关心的是“还能不能继续生成”和“怎么买完马上继续用”，而不是账户体系。
3. 前台应隐藏“管理员”“客户账号”等内部概念。
4. 现有 access code 可在系统内部继续作为兑换码使用，前台只改变表达和交互。

## 5. 用户状态

前台统一按以下状态组织体验：

1. `匿名试用中`
2. `试用即将用完`
3. `试用已用完，待兑换`
4. `已兑换 / 有可用额度`

## 6. 关键体验要求

### 6.1 状态优先级

优先显示：

- 试用剩余次数
- 可用额度
- 是否还能继续生成

弱化显示：

- 是否登录
- 内部账号结构

### 6.2 试用刷新规则

必须明确告知用户：

- 试用额度按 7 天窗口刷新
- 如果服务端返回具体刷新时间，则优先显示具体时间
- 如果没有具体刷新时间，至少显示“每 7 天刷新一次”

### 6.3 购买 / 兑换主入口

所有“登录”心智统一改为“购买 / 兑换”。

用户侧统一使用以下语言：

- 购买额度
- 输入兑换码
- 继续使用

不使用以下语言作为主表达：

- 客户登录
- 管理员分配账号
- 上游 / 网关 / 访问码登录

## 7. 前台交互规格

### 7.1 顶部入口

当前顶部 `登录` 按钮改为：

- `购买/兑换`

### 7.2 顶部状态文案

匿名状态建议显示：

- `试用剩余 2 次`
- `试用将于 YYYY-MM-DD HH:mm 刷新`

或在没有具体时间时：

- `试用剩余 2 次`
- `试用每 7 天刷新一次`

已兑换 / 已认证状态建议显示：

- `可用额度 50`

### 7.3 试用耗尽时的主提示

提交被拦截时不再提示“请登录客户账号”，而是：

- `试用已用完，购买额度后可继续生成`

### 7.4 兑换弹窗

当前 Auth Modal 重构为“继续使用”弹窗。

推荐结构：

#### 区块 A：兑换

- 标题：`我已有兑换码`
- 输入项：`兑换码`
- 主按钮：`立即兑换`
- 辅助文案：`兑换成功后，额度会直接加到当前浏览器。`

#### 区块 B：购买

- 标题：`我还没有兑换码`
- 主入口：`自助购买兑换码`
- 次入口：`联系客服购买`
- 辅助文案：`购买后会获得兑换码，输入后即可继续使用。`

### 7.5 购买入口展示原则

当前阶段购买渠道只体现为两类用户动作：

1. `自助购买`
2. `联系客服购买`

前台不直接平铺内部运营描述，例如：

- 抖音付款
- 微信转账
- 发卡网站

这些属于渠道实现，不应成为主产品语言。

## 8. 后台与数据规格

### 8.1 最小数据策略

复用现有：

- customer
- session
- remaining credits
- access code hash

最小新增能力：

- 支持用户仅通过兑换码建立会话

### 8.2 会话语义

兑换成功后：

- 当前浏览器进入 authenticated 状态
- 顶部状态更新为额度视图
- 生成能力立即恢复

### 8.3 兼容性要求

如仍需兼容历史 `邮箱 + 访问码` 路径：

- 可以在后端保留旧逻辑
- 但前台默认优先走兑换码路径

## 9. 文案原则

本次流程中的文案必须满足：

1. 先告诉用户还能不能继续用
2. 再告诉用户下一步怎么做
3. 不暴露后台结构
4. 不让用户理解复杂账户模型

## 10. 验收标准

### 10.1 用户体验验收

- 用户在试用中可以明确看到剩余试用次数
- 用户能知道试用刷新规则
- 试用耗尽后，主路径明确进入购买 / 兑换，而不是登录
- 用户看到兑换码输入入口时，不需要理解邮箱和客户账号

### 10.2 技术验收

- 不破坏现有匿名试用能力
- 不破坏现有已认证用户的生成能力
- 兑换成功后会话正确建立
- 构建和测试通过

### 10.3 商业验收

- 用户从“试用已用完”到“知道如何继续使用”的路径明显更短
- 客服不必再先解释“登录客户账号”的内部逻辑
