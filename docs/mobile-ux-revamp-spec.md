## 1. 前提与假设

本规格基于以下前提推进：

1. 本次改造只针对移动端主应用，不重做桌面端。
2. 本次改造不调整后端接口、额度逻辑、生成流程语义。
3. 首次打开页面时，案例指引必须保留，不能改成只剩输入框。
4. `生成图像` 是移动端主操作按钮，视觉权重和路径优先级都应高于其他操作。
5. 当前项目仍应保留既有产品气质，不做整站视觉翻新。

## 2. 背景

当前移动端界面已经具备完整能力，但在真实手机视口下存在明显的信息架构问题：

- 首次进入时，案例区信息量过大，首屏更像长列表而不是引导页。
- 底部输入台默认展开层级偏高，案例区和输入区同时争夺首屏注意力。
- `生成图像` 按钮在移动端虽然可用，但主操作感偏弱。
- 顶部状态、帮助、设置和购买入口同时堆在一行，头部较拥挤。
- 弹层和详情区仍偏桌面思路，在手机上不够顺手。

问题不在于“功能缺失”，而在于“首次成功体验不够顺”。用户第一次进入时，应该先被正确引导，再自然进入输入和生成，而不是自己判断下一步在哪里。

## 3. 目标

本次移动端改造要完成以下目标：

1. 保留案例指引作为首次打开时的主内容。
2. 让用户在看完案例后，能自然进入“修改少量内容并生成”的下一步。
3. 强化 `生成图像` 作为主操作按钮的识别度和点击意图。
4. 收敛移动端信息密度，让头部、案例区、输入区、历史区层级更清楚。
5. 在不破坏当前工作流的前提下，提升首轮成功体验与持续生成体验。

## 3.1 P0 范围

本轮实现只做最小但高收益的移动端改造，范围收敛为：

1. 首次进入时的案例区重排
2. 案例点击后的过渡状态
3. 底部输入台默认态精简
4. `生成图像` 主按钮强化
5. 头部移动端挤压问题的最小修正

以下内容暂不纳入本轮：

- `HelpModal` 结构重做
- `DetailModal` 改成 sheet
- 历史卡片信息结构重排
- 多选交互重构

## 4. 非目标

本次不做：

- 整站重设计
- 桌面端交互重构
- 历史任务系统能力改写
- 购买 / 兑换流程语义调整
- 新增后端能力或接口字段
- 全量替换设计语言

## 5. 用户场景

### 5.1 首次进入

用户第一次打开页面时，预期路径应为：

1. 先看到案例指引
2. 选择一个接近需求的案例
3. 明确知道“下一步是直接生成”还是“先上传参考图”
4. 用最少改动开始第一张图

### 5.2 已有历史记录后继续生成

用户已有任务记录时，预期路径应为：

1. 快速看到最近结果
2. 继续通过底部输入台发起下一轮生成
3. 案例入口作为辅助，不抢占主视觉

### 5.3 生成中

用户提交后，预期路径应为：

1. 明确知道请求已经发出
2. 一眼看到当前正在生成哪一条任务
3. 不需要在筛选区、案例区和输入区之间自行找焦点

## 6. 关键体验原则

### 6.1 案例指引不是附属功能

首次打开时，`FirstImageStarter` 不是可选装饰，而是首轮成功体验的一部分。  
但它应承担“引导开始”的职责，而不是变成长篇内容列表。

### 6.2 主操作必须更强

移动端主路径中的主操作是 `生成图像`。  
它在视觉上必须比筛选、帮助、设置、附件等次级动作更明确。

### 6.3 复杂参数按需展开

尺寸、质量、数量等参数属于高级能力。  
移动端默认不应与首次案例指引和主操作同时抢首屏。

### 6.4 选完案例后必须有明确过渡

案例被点击后，页面必须明确表达当前状态：

- 已填入案例
- 是否还需要补参考图
- 现在是否可以直接生成

不能要求用户自己判断“下一步该点哪里”。

## 7. 交互规格

### 7.1 空状态首页

当没有历史任务时，移动端首页应采用“两步式首启结构”：

#### 区块 A：案例引导区

保留在首屏顶部，作为首次进入时的主内容。

要求：

- 仍使用 `FirstImageStarter`
- 移动端改为更紧凑的案例卡形态
- 每张卡只展示：
  - 标题
  - 适用场景
  - 等级标签
  - 是否需要参考图
  - 明确动作文案
- 不在卡片里直接铺大段 prompt 正文

#### 区块 B：输入与生成区

仍固定在底部，但默认态要收口。

要求：

- 输入框清晰可见
- `生成图像` 按钮在视觉上是唯一主按钮
- 高级参数默认收起
- 如果案例要求参考图，应清楚提示“请先上传参考图”

### 7.2 案例卡点击后的状态

点击案例后，界面必须进入明确的下一步状态。

要求：

- 自动把案例 prompt 填入输入框
- 给出清晰反馈文案
- 如果该案例可直接生成，应强化 `生成图像`
- 如果该案例依赖参考图，应优先强调 `添加参考图`

可接受的表达方向：

- `已填入案例，可直接生成`
- `已填入案例，请先上传参考图`

### 7.3 底部输入台

移动端 `InputBar` 调整为“默认精简、按需展开”。

默认展示：

- prompt 输入框
- `添加参考图`
- `生成图像`

默认收起：

- 尺寸
- 质量
- 格式
- 压缩率
- 审核
- 数量

展开方式要求：

- 有明确的展开入口或可理解的折叠提示
- 不能只依赖隐式拖拽理解

### 7.4 主按钮规格

`生成图像` 按钮要求：

- 在移动端默认态中视觉最突出
- 宽度、颜色、对比度明显优于次级按钮
- 禁用态仍需保持“这是主按钮”的识别度
- 在已满足提交条件时，一眼能看出可以点击

`添加参考图` 按钮要求：

- 作为次主操作存在
- 当案例必须依赖参考图时，其提示优先级上升，但视觉权重仍不超过主按钮

### 7.5 顶部头部

移动端头部应收口为更紧凑的结构。

必须保留：

- 品牌识别
- 试用 / 额度状态
- `购买/兑换`

建议弱化或收纳：

- `操作指南`
- `设置`

要求：

- 不出现品牌和状态互相挤压
- 不出现头部多行换行
- 保持首屏垂直空间尽量节省

### 7.6 历史任务态

当已有历史任务时：

- 历史记录恢复为主内容区
- 案例入口改为次级入口
- 底部输入台继续承担持续生成职责

移动端不应在已有任务时继续默认展示完整首启案例列表。

### 7.7 生成中状态

提交成功后，移动端界面必须优先突出“当前正在生成的任务”。

要求：

- 新任务卡尽量出现在用户当前可见区域上方
- 加载中的状态文案清晰
- 不让筛选区、案例入口、底部输入栏同时抢第一焦点

### 7.8 帮助与详情弹层

移动端弹层优化保留为后续方向，不进入本轮 P0 实现。

- 当前要求：

- 不因为本轮移动端改造而破坏现有帮助弹层和详情弹层
- 如需顺手修正明显的移动端间距问题，可做微调

## 9. 实施顺序

建议按以下顺序实现：

1. `FirstImageStarter` 移动端首启重排
2. `TaskGrid` 空状态与历史态切换逻辑调整
3. `InputBar` 移动端精简态与主按钮强化
4. `Header` 收口
5. 真实移动视口回归截图与交互验证

## 9.1 实现切片

为避免引入过渡冗余，本轮拆成 3 个简单切片：

1. `Starter`
   - 压缩移动端案例卡信息密度
   - 保留案例指引主内容地位
2. `Composer`
   - 收起移动端高级参数默认态
   - 强化 `生成图像` 按钮
   - 明确案例点击后的下一步提示
3. `Header`
   - 只做移动端最小收口
   - 不改购买 / 兑换语义

## 11. 验证策略

### 11.1 手动验证

至少在移动端视口下验证以下场景：

1. 首次打开，无历史记录
2. 点击可直接生成的案例
3. 点击需要参考图的案例
4. 手动输入 prompt 后可成功进入可提交状态
5. 提交后能看到生成中任务
6. 打开帮助弹层
7. 打开任务详情

建议基准视口：

- `390 x 844`
- `393 x 852`

### 11.2 回归要求

- 不破坏桌面端布局
- 不破坏案例填入能力
- 不破坏底部输入台固定工作流
- 不破坏已有历史任务浏览与详情能力
- 不引入新的复杂过渡状态

## 12. 边界

### Always

- 保留首次打开时的案例指引
- 保持 `生成图像` 为移动端主操作
- 改动后用真实移动视口截图复查
- 尽量只动移动端主路径相关组件

### Ask First

- 调整桌面端主要布局
- 删除现有案例内容
- 修改购买 / 兑换主路径
- 引入新的 UI 依赖

### Never

- 为了移动端改造而重写整站设计语言
- 删除当前持续生成的底部输入台模式
- 把首次打开改成没有案例引导的空输入页

## 13. 成功标准

本次改造完成后，应满足：

1. 首次打开时，案例指引仍是主内容，但首屏信息密度明显下降。
2. 用户点击案例后，能明确知道下一步是直接生成还是先上传参考图。
3. `生成图像` 在移动端的视觉权重明显提升。
4. 底部输入台默认态更简洁，不再与案例区严重抢首屏注意力。
5. 已有历史任务时，历史区重新成为主内容，案例入口退居次级。
6. 在真实移动端截图中，头部、主内容区、底部输入台层级清晰。

## 14. Open Questions

当前先保留以下问题，进入实现前如有需要可再细化：

1. 案例区在移动端采用横滑、纵向紧凑列表，还是分段折叠，哪个更符合当前页面节奏。
2. 高级参数的展开入口是文字按钮、图标按钮，还是保留拖拽手柄并补充明确文案。
3. 帮助和设置是否合并到统一更多菜单，还是保持双图标但降低视觉权重。
