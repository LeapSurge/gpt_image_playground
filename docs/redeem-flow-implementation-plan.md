# 购买 / 兑换流程实现计划

状态：Draft v0.1  
对应规格：[redeem-flow-spec.md](/D:/project/gpt_image_playground/docs/redeem-flow-spec.md)

## 1. 实现原则

本次按最小闭环实现：

- 不引入支付系统
- 不重做用户模型
- 不大改页面结构
- 前台改心智，后台复用已有结构

## 2. 范围

### 2.1 前端范围

- 顶部入口与状态文案
- 试用耗尽提示
- Auth Modal 重构为购买 / 兑换弹窗
- 试用刷新规则的用户可见说明

### 2.2 后端范围

- 新增或改造“按兑换码建立会话”能力
- 保留现有 session / customer / credits 结构
- 必要时在 session 接口中继续提供 trial reset 信息

### 2.3 文档范围

- 产品规格
- 实现计划
- 如有必要，补充手工测试清单

## 3. 分阶段计划

### Phase 1：后端最小能力

入口条件：

- 已确认当前阶段使用“站外支付 + 兑换码”

动作：

1. 评估复用 access code 作为兑换码是否足够
2. 增加“仅凭兑换码建立会话”的最小 API 能力
3. 保留现有 customer/session 存储模型
4. 确保匿名试用 session 与兑换后 session 切换正确

退出条件：

- 后端可支持前台只输入兑换码完成会话建立

验证：

- 构建通过
- 测试通过
- 手工调用新路径可成功返回 authenticated session

### Phase 2：前台购买 / 兑换重构

入口条件：

- 后端兑换能力可用，或已明确可兼容的过渡方案

动作：

1. 将顶部“登录”入口改为“购买/兑换”
2. 将 Auth Modal 从“客户登录”改为“继续使用”
3. 将主输入从“邮箱 + 访问码”收敛为“兑换码”为主
4. 补充“自助购买 / 联系客服购买”入口
5. 在匿名状态下展示试用刷新规则

退出条件：

- 用户在前台不再需要理解“客户账号登录”

验证：

- 匿名状态能看到试用次数与刷新规则
- 试用耗尽后主提示正确
- 兑换成功后额度与生成状态立即恢复

### Phase 3：文案与状态收口

入口条件：

- 基本流程已跑通

动作：

1. 清理仍残留的“登录 / 客户 / 管理员”用户侧语言
2. 统一错误提示、成功提示、阻断提示
3. 明确兑换成功到账反馈

退出条件：

- 同一流程中不再出现相互冲突的心智语言

验证：

- 抽查 Header、InputBar、AuthModal、Settings 等关键位置文案一致

## 5. 风险与控制

### 风险 A：把“兑换码”做成了全新用户系统

控制：

- 严格复用现有 customer/session
- 只新增最小认证入口

### 风险 B：前台仍然带着后台语言

控制：

- 文案统一改成“购买 / 兑换 / 继续使用”

### 风险 C：试用刷新规则未说清，用户误解

控制：

- 匿名状态下必须有明确说明
- 优先显示具体 resetAt
- 没有 resetAt 时显示“每 7 天刷新一次”

### 风险 D：购买入口信息不完整

控制：

- 前台先支持“自助购买 / 联系客服购买”两个稳定占位
- 渠道链接缺失时也不阻断基础兑换流程

## 6. 验证清单

### 用户侧

- 新用户首次进入能看到试用信息
- 新用户知道试用不是永久的
- 试用耗尽后知道如何继续
- 输入兑换码后能继续生成

### 运营侧

- 不需要先向用户解释“客户账号登录”
- 用户提问更聚焦于购买和兑换本身

### 工程侧

- `npm run build`
- `npm test`
- 手工验证匿名试用、兑换、已兑换生成三条路径
