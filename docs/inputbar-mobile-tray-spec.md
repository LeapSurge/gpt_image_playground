# InputBar 移动端托盘规格

状态：Draft v0.1  
适用范围：`<= 640px` 的移动端 `InputBar`  
关联文档：
- [mobile-ux-revamp-spec.md](/D:/project/gpt_image_playground/docs/mobile-ux-revamp-spec.md)
- [product-design-guidelines.md](/D:/project/gpt_image_playground/docs/product-design-guidelines.md)

## 1. 背景

当前移动端 `InputBar` 已经具备：

- prompt 输入
- 参考图上传
- 参数调整
- 生成提交

但它的表达方式不够统一：

- 视觉上像底部托盘
- 交互上却不像真正可展开的托盘
- 输入框和参数栏像两块并列卡片，而不是一个完整工作面板

结果是用户很难一眼理解：

1. 这是固定底部的创作区
2. 这是可以展开/收起的托盘
3. 输入框、参数、图片和生成按钮之间谁是主、谁是次

## 2. 目标

本规格要解决的问题只有一个：

`InputBar` 必须成为一个语义清晰、可理解、可展开的移动端创作托盘。

具体目标：

1. 让用户明确知道这是一个底部托盘，而不是普通静态表单
2. 让托盘在收起态和展开态都成立
3. 让输入框成为主工作区，参数区成为次级工具区
4. 让 `生成图像` 保持主操作按钮地位

## 3. 非目标

本次不做：

- 桌面端 `InputBar` 重构
- 提交流程逻辑调整
- 参数语义调整
- 新增复杂动画
- 新增额外步骤或引导气泡

## 4. 核心定义

移动端 `InputBar` 定义为：

`固定底部创作托盘`

它不是：

- 普通输入框
- 参数面板
- 独立弹窗

它应该同时承担：

- 快速发起创作
- 按需展开高级控制
- 保持持续生成工作流

## 5. 状态模型

### 5.1 收起态

收起态是默认态。

它的职责：

- 让用户快速看到可以开始创作
- 占用尽量少的垂直空间
- 保留当前内容状态感知

收起态应显示：

- 明确的托盘把手
- 单行 prompt 入口或摘要
- `添加参考图`
- `生成图像`
- 一行轻量参数摘要

收起态不应显示：

- 完整参数网格
- 大块输入区
- 冗余说明文字

### 5.2 展开态

展开态是工作态。

它的职责：

- 承担主要输入编辑
- 提供参数调整能力
- 展示已添加图片

展开态应显示：

- 更大的输入框
- 参考图缩略图
- 参数区
- 底部主操作区

## 6. 交互规则

### 6.1 展开触发

以下动作应展开托盘：

- 点击输入区
- 点击把手
- 上滑托盘

### 6.2 收起触发

以下动作应收起托盘：

- 点击把手
- 下滑托盘

是否允许点击空白自动收起，后续再定，本轮先不强制。

### 6.3 状态连续性

收起与展开切换时，必须保留：

- 当前 prompt
- 已添加参考图
- 当前参数

用户不能产生“内容消失”感。

## 7. 视觉层级

### 7.1 主层级

输入框是托盘中的主视觉区。

要求：

- 比参数栏更大
- 比参数栏更亮
- 比参数栏更像“工作面”

### 7.2 次层级

参数区是次级工具区。

要求：

- 更轻
- 更小
- 更工具化

它不应与输入框形成两个平级大卡片。

### 7.3 主按钮

`生成图像` 是托盘中的主操作按钮。

要求：

- 默认比 `添加参考图` 更强
- 在收起态也保持主按钮地位
- 在可提交时一眼可点

### 7.4 次按钮

`添加参考图` 是次主操作。

要求：

- 快速可达
- 视觉强度低于 `生成图像`

### 7.5 把手

把手的作用是提示“这是托盘，可展开/收起”。

要求：

- 明显但不抢戏
- 不应做成另一个主按钮

## 8. 结构原则

### 8.1 一体化托盘

移动端 `InputBar` 必须是一整块托盘，而不是：

- 输入框一张卡
- 参数栏一张卡
- 按钮区另一张卡

这三部分应属于同一个底部面板内部。

### 8.2 收起态要明显更矮

如果视觉上看起来像托盘，收起态就必须明显更矮。  
否则用户会认为它只是一个固定卡片，而不是可操作的托盘。

### 8.3 展开态要像工作台

展开后，用户必须直觉感受到：

- 这是当前的主要工作区域
- 可以在这里连续输入、调参、提交

## 9. 文案与反馈

### 9.1 收起态摘要

收起态应保留一行轻量摘要，用来说明当前状态，例如：

- `先写主体，再补风格和场景`
- `已填入案例，可继续生成`
- `已填入案例，请先上传参考图`
- `当前 自动 · PNG · 1 张`

### 9.2 案例填入后的反馈

点击案例后，托盘需要明确表达下一步，而不是只把文字塞进输入框。

至少应让用户知道两件事：

1. 案例已填入
2. 现在是可以直接生成，还是需要先上传参考图

## 10. 设计方向

本组件建议采用：

`iOS sheet + 创作工具栏`

但实现上应克制：

- 不做复杂弹簧动画
- 不做多层级面板
- 不做重型手势系统

重点不是炫技，而是让托盘语义成立。

## 11. 实现切片

建议按以下顺序实现：

1. 重构移动端托盘 DOM 层级
2. 加入明确的收起态 / 展开态
3. 把参数区嵌回托盘内部
4. 调整输入框、参数区、按钮区的层级
5. 用真实移动端截图回归

## 12. 验证标准

完成后应满足：

1. 用户能一眼看出这是底部托盘
2. 托盘收起和展开都成立
3. 输入框明显大于参数栏
4. 参数栏明显小于输入框
5. `生成图像` 在两个状态下都保持主按钮地位
6. 用户不会再感到“像托盘但不能拉开”

## 13. 边界

### Always

- 保持 `InputBar` 固定底部工作流
- 保持输入、上传、提交逻辑不变
- 改动后用真实移动端截图验证

### Ask First

- 改动桌面端布局
- 调整参数语义
- 增加新的引导层

### Never

- 继续保留“像托盘但交互不成立”的中间状态
- 把输入框和参数栏继续做成两个平级主卡片
- 用复杂动画掩盖结构问题
